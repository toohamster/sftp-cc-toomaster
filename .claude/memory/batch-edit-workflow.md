# 批量修改工作流程（操作规范）

## 教训来源

**问题**：在修改多个相似文件时（如 24 个章节的授权声明），只做零散修复，没有系统性检查和验证，导致连续遗漏多个文件。

**连续犯错的原因**：
1. 急于"修复"，而不是先"理解"问题范围
2. 过度依赖用户指令，没有主动思考
3. 没有建立"检查清单"习惯，假设"没报错=没问题"
4. 目录结构没搞清楚就创建文件

---

## 操作规范

### 原则：出方案 → 核对方案 → 再执行

**禁止**在未列出完整清单前就动手修改！

---

### 步骤 1：出方案

在动手之前，必须先明确并向用户汇报：

| 项目 | 内容 |
|------|------|
| **修改范围** | 列出完整文件清单（例如：24 个章节 = 8 章 × 3 语言） |
| **当前状态** | 哪些已完成？哪些未完成？哪些有错误？ |
| **修改内容** | 具体要改什么？（示例说明） |
| **目标格式** | 修改后应该是什么样子？（标准模板） |

```bash
# 检查命令示例
for f in myskillNotes/*/chapter-*.md; do echo "=== $f ==="; tail -10 "$f"; done
```

---

### 步骤 2：核对方案

在执行前，必须等待用户确认：

- [ ] 文件清单是否完整？
- [ ] 问题描述是否准确？
- [ ] 目标格式是否正确？
- [ ] 用户确认可以继续？

**禁止**在用户确认前就动手！

---

### 步骤 3：执行修改

确认方案无误后：

1. 按逻辑分组处理（例如：en → zh-cn → ja）
2. 每处理完一组，立即验证
3. 记录已完成的文件

---

### 步骤 4：全面验证

修改完成后，必须执行：

```bash
# 再次检查所有文件
for f in myskillNotes/*/chapter-*.md; do echo "=== $f ==="; tail -10 "$f"; done

# 提交 git status 清单给用户确认
git status
```

验证清单：
- [ ] 所有文件格式统一
- [ ] 所有链接路径正确
- [ ] 没有遗漏任何文件
- [ ] git status 清单已提交用户确认

---

## 适用场景

此工作流程适用于所有批量修改任务：
- 批量修改章节 footer
- 批量更新多语言文件
- 批量修复引用路径
- 批量添加/删除内容
- 任何涉及多个相似文件的修改任务

---

## 核心检查命令

```bash
# 统计文件总数
find myskillNotes -name "chapter-*.md" | wc -l

# 检查所有章节 footer
for f in myskillNotes/*/chapter-*.md; do echo "=== $f ==="; tail -10 "$f"; done

# 检查特定模式
grep -r "pattern" myskillNotes/*/ --include="*.md"
```

---

**记住：慢就是快。先核对，再执行，一次做对。**
