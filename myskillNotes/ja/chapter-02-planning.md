## 2.1 要件分析

### 2.1.1 ペインポイントから出発

sftp-cc 開発の元の動機を記録しましょう：

> 「PhpStorm を使用していたとき、組み込みの SFTP 拡張は自動的にファイルをサーバーに同期してくれました。Claude Code に切り替えた後、この機能を失いました — Claude がコードを変更するたびに、テストサーバーで手動でプルする必要があり、非常に非効率的でした。」

このペインポイントを分解しましょう：

| ペインポイント | 影響 | 深刻度 |
|---------------|------|--------|
| 手動同期が必要 | 時間の浪費 | 高 |
| 頻繁なコンテキストスイッチ | 集中力の低下 | 中 |
| 人的ミスのリスク | 本番環境の問題 | 高 |
| 非効率的なワークフロー | 生産性の低下 | 高 |

### 2.1.2 要件リスト

要件を優先度と共にリストしましょう：

| 要件 | 優先度 | 説明 | なぜ必要か |
|------|--------|------|------------|
| サーバーにファイルをアップロード | ⭐⭐⭐ | コア機能 | 主な目的 |
| 増分アップロード | ⭐⭐⭐ | 変更されたファイルのみをアップロード | 時間と帯域の節約 |
| 自動秘密鍵バインド | ⭐⭐ | 設定を簡素化 | ユーザー体験の向上 |
| 権限修正 | ⭐⭐ | chmod 600 | セキュリティ要件 |
| 多言語サポート | ⭐ | 国際化 | より広いユーザーベース |
| 設定初期化 | ⭐⭐ | 対話式セットアップ | 初心者フレンドリー |
| 公開鍵デプロイ | ⭐ | SSH 認証セットアップ | サーバー準備 |

### 2.1.3 機能バウンダリ

**やること**：
- ✅ SFTP を使用したファイルアップロード
- ✅ Git ベースの増分検出
- ✅ SSH 鍵認証
- ✅ 単一サーバーデプロイ
- ✅ 対話式設定初期化

**やらないこと**：
- ❌ ファイルダウンロード（片方向同期のみ）
- ❌ リアルタイムファイル変更監視（複雑すぎる）
- ❌ マルチサーバー同時デプロイ（スコープ外）
- ❌ FTP/S 認証（SSH 鍵のみ）
- ❌ GUI インターフェース（CLI のみ）

**なぜ「やらないこと」を定義するか**：
1. スコープクリープを防止
2. 開発リソースを集中
3. ユーザー期待を管理
4. 将来的な拡張のために残しておく

## 2.2 ユーザーペルソナ

### 2.2.1 主要ペルソナ：フルスタック開発者

**名前**: 田中太郎
**職業**: フルスタック Web 開発者
**技術スタック**: PHP, JavaScript, MySQL
**使用ツール**: PhpStorm, VS Code, Claude Code

**シナリオ**:
> 太郎は Claude Code を使用して迅速にコードを変更しています。変更後、テストサーバーにデプロイして確認する必要があります。以前は PhpStorm の自動同期を使用していましたが、Claude Code では手動で FTP クライアントを開く必要があります。

**sftp-cc が解決する問題**:
- 自然言語でデプロイ：「コードをサーバーに同期」
- 自動で変更ファイルのみをアップロード
- 設定は一度きり、その後自動化

### 2.2.2 二次ペルソナ：DevOps エンジニア

**名前**: 鈴木花子

**シナリオ**:
> 花子は複数のプロジェクトを管理しており、それぞれが異なるサーバーにデプロイされます。設定管理とデプロイスクリプトの維持に時間を費やしています。

**sftp-cc が提供する価値**:
- 複数プロジェクト設定のサポート
- スクリプトによる自動化統合
- 明確な設定ファイル形式

## 2.3 機能設計

### 2.3.1 コア機能モジュール

```
sftp-cc
├── 設定初期化モジュール (sftp-init.sh)
│   ├── 設定ディレクトリ作成
│   ├── 設定テンプレートコピー
│   └── 対話式パラメータ入力
│
├── 秘密鍵管理モジュール (sftp-keybind.sh)
│   ├── 秘密鍵スキャン
│   ├── 権限修正 (chmod 600)
│   └── 設定にパスを書き込み
│
├── 公開鍵デプロイモジュール (sftp-copy-id.sh)
│   ├── 公開鍵確認
│   ├── サーバーにデプロイ
│   └── 認証テスト
│
└── ファイルアップロードモジュール (sftp-push.sh)
    ├── 変更ファイル検出
    ├── SFTP バッチファイル構築
    └── アップロード実行
```

### 2.3.2 アップロードモード設計

各モードの使用ケースを定義：

| モード | コマンド | 使用ケース | 例 |
|------|---------|-------------|-----|
| 増分アップロード | `sftp-push.sh` | 通常使用、変更のみをデプロイ | 機能開発後 |
| 完全アップロード | `sftp-push.sh --full` | 初回デプロイ、全ファイルを送信 | 新規プロジェクト |
| 指定ファイル | `sftp-push.sh file.php` | 単一ファイルの修正をデプロイ | バグ修正後 |
| 指定ディレクトリ | `sftp-push.sh -d src/` | 特定ディレクトリのみをデプロイ | モジュール更新 |
| プレビューモード | `sftp-push.sh -n` | 実際のアップロード前に確認 | 安全確認 |

### 2.3.3 増分検出ロジック

Git の変更を 4 つのカテゴリで検出：

```bash
# 1. コミット済み変更（最後のコミット以降）
git diff --name-only --diff-filter=ACMR <last_hash> HEAD
# ACMR: Added, Copied, Modified, Renamed

# 2. ステージング済み変更（インデックス）
git diff --cached --name-only --diff-filter=ACMR
# git add 済みだが未コミットのファイル

# 3. ワークキングディレクトリ変更（未ステージング）
git diff --name-only --diff-filter=ACMR
# 編集中のファイル

# 4. 未トラックファイル（Git 管理外）
git ls-files --others --exclude-standard
# .gitignore にない未管理ファイル
```

**なぜ 4 つすべて必要か**：
- ユーザーは「すべての変更」をデプロイしたい
- コミット済み、未コミット、未管理を区別しない
- 網羅的なカバレッジでユーザーミスを防止

## 2.4 ディレクトリ構造計画

### 2.4.1 最終構造

```
sftp-cc/
├── .claude-plugin/
│   └── marketplace.json        # Plugin marketplace 設定
├── skills/
│   └── sftp-cc/
│       └── SKILL.md            # Skill 定義（コア）
├── scripts/
│   ├── sftp-init.sh            # 設定初期化
│   ├── sftp-keybind.sh         # 秘密鍵バインド
│   ├── sftp-copy-id.sh         # 公開鍵デプロイ
│   ├── sftp-push.sh            # ファイルアップロード
│   └── i18n.sh                 # 多言語サポート
├── templates/
│   └── sftp-config.example.json # 設定テンプレート
├── skill.md                     # 手動インストール Skill 定義
├── install.sh                   # 手動インストールスクリプト
├── README.md                    # 英語ドキュメント
├── README_CN.md                 # 中国語ドキュメント
├── README_JP.md                 # 日本語ドキュメント
├── SPEC.md                      # 技術仕様
└── CLAUDE.md                    # 開発ガイド
```

### 2.4.2 各ディレクトリの目的

| ディレクトリ | 目的 | 含まれるファイル |
|-------------|------|-----------------|
| `.claude-plugin/` | Plugin marketplace 設定 | marketplace.json |
| `skills/` | Skill 定義 | SKILL.md |
| `scripts/` | 実行可能スクリプト | *.sh |
| `templates/` | 設定テンプレート | *.example.json |
| (ルート) | ドキュメント、インストール | README, install.sh |

### 2.4.3 2 つのインストール方法

Plugin には 2 つのインストール方法があります：

| 項目 | Plugin インストール | 手動インストール |
|------|---------------------|------------------|
| **パス** | `~/.claude/plugins/marketplaces/sftp-cc/` | `.claude/skills/sftp-cc/` |
| **方法** | `/plugin marketplace add <URL>` | `bash install.sh <path>` |
| **更新** | 自動 | 手動 |
| **推奨** | ✅ はい | 旧バージョン互換 |
| **SKILL.md 変数** | `${CLAUDE_PLUGIN_ROOT}` | 相対パス |

**なぜ両方をサポートするか**：
1. Plugin marketplace は新しい標準
2. 手動インストールは既存ユーザーと互換
3. 移行期間中の柔軟性

## 2.5 設定ファイル設計

### 2.5.1 sftp-config.json 完全な例

```json
{
  "host": "example.com",
  "port": 22,
  "username": "deploy",
  "remote_path": "/var/www/html",
  "local_path": ".",
  "private_key": "",
  "language": "ja",
  "excludes": [
    ".git",
    ".claude",
    "node_modules",
    ".env",
    ".DS_Store"
  ]
}
```

### 2.5.2 フィールド詳細

| フィールド | 型 | 必須 | デフォルト | 説明 | 例 |
|-------|------|------|----------|-------------|-----|
| `host` | 文字列 | はい | - | SFTP サーバーアドレス | `"example.com"` |
| `port` | 数値 | いいえ | `22` | SFTP ポート | `22` |
| `username` | 文字列 | はい | - | ログインユーザー名 | `"deploy"` |
| `remote_path` | 文字列 | はい | - | リモートデプロイ先パス | `"/var/www/html"` |
| `local_path` | 文字列 | いいえ | `"."` | ローカルソースディレクトリ | `"."` |
| `private_key` | 文字列 | いいえ | `""` | 秘密鍵パス（自動入力） | `""` |
| `language` | 文字列 | いいえ | `"en"` | インターフェース言語 | `"ja"` |
| `excludes` | 配列 | いいえ | 下記参照 | 除外ファイル/ディレクトリ | `[".git"]` |

### 2.5.3 設計上の決定

**なぜ JSON か**：
- ✅ Claude Code がネイティブに解析
- ✅ 人間に読みやすい
- ✅ コメント以外はシンプルな構文
- ✅ ほぼすべてのプログラミング言語でサポート

**なぜ `excludes` 配列か**：
- 一般的に除外されるファイルはプロジェクト間で類似
- デフォルトを提供し、ユーザーカスタマイズを許可
- 配列形式は拡張可能

**デフォルトの除外パターン**：
```json
"excludes": [
  ".git",           // Git リポジトリ
  ".claude",        // Claude 設定
  "node_modules",   // Node.js 依存
  ".env",           // 環境変数（機密）
  ".DS_Store"       // macOS システムファイル
]
```

## 2.6 トリガーワード設計

### 2.6.1 設計原則

良いトリガーワードの特徴：

1. **自然言語**: ユーザーが実際に言うフレーズ
   - ✅ 「コードをサーバーに同期」
   - ❌ 「sftp_push_execute」

2. **多言語カバレッジ**: 主要言語をカバー
   - 英語、中国語、日本語

3. **曖昧さ回避**: 他の機能と競合しない
   - ✅「sync code to server」
   - ❌「push」（git push と競合）

### 2.6.2 アップロード系トリガーワード

```
英語：
- "sync code to server"
- "upload to server"
- "deploy code"
- "push to server" (注：git push と明確化)
- "sync changes"

中国語：
- "同步代码到服务器"
- "上传到服务器"
- "部署代码"
- "推送到服务器"

日本語：
- "サーバーに同期する"
- "デプロイする"
- "コードをアップロード"
- "変更を反映"
```

### 2.6.3 秘密鍵バインド系トリガーワード

```
英語：
- "bind sftp private key"
- "bind ssh key"
- "setup sftp authentication"

中国語：
- "绑定 SFTP 私钥"
- "绑定私钥"
- "设置 SFTP 认证"

日本語：
- "秘密鍵をバインドする"
- "SSH 鍵をバインドする"
- "SFTP 認証を設定"
```

### 2.6.4 トリガーしないワード

以下のワードは意図的にトリガーとして扱いません：

| ワード | 理由 |
|-------|------|
| "push" | git push と競合 |
| "推送" | 同上 |
| "プッシュ" | 同上 |
| "commit" | git commit と競合 |
| "deploy"（単独）| 曖昧すぎる、他のデプロイ手段と競合 |

## 2.7 技術選択

### 2.7.1 なぜ Shell を選ぶのか

スクリプト言語の比較：

| 項目 | Shell | Python | Node.js |
|------|-------|--------|---------|
| 外部依存 | なし（システム標準） | pip が必要 | npm が必要 |
| システム互換性 | 全 UNIX/Linux システムに標準搭載 | インストール必要 | インストール必要 |
| 学習曲線 | 低い（基本コマンド） | 中 | 中 |
| 実行速度 | 速 | 中 | 中 |
| 開発速度 | 速 | 速 | 速 |
| 移植性 | 高い | 高い | 高い |

### 2.7.2 外部依存ゼロの原則

sftp-cc は次の原則に従います：

**原則**: システム標準コマンドのみを使用し、外部パッケージに依存しない

**理由**:
1. **インストールの手間**: ユーザーは追加パッケージをインストールする必要がない
2. **互換性**: ほぼすべての UNIX/Linux システムで動作
3. **信頼性**: 外部パッケージのバージョン変更の影響を受けない
4. **軽量**: 依存関係が少なく、トラブルシューティングが容易

**実装例**：

```bash
# ❌ 依存あり：jq を使用
value=$(jq -r '.host' config.json)

# ✅ 依存なし：純粋な Shell
value=$(grep '"host"' config.json | sed 's/.*: *"\([^"]*\)".*/\1/')
```

**使用するシステムコマンド**：
- `sftp` — SFTP クライアント（OpenSSH）
- `git` — バージョン管理
- `grep`, `sed` — テキスト処理
- `mktemp` — 一時ファイル作成
- `stat` — ファイル情報取得
- `chmod` — 権限変更

## 2.8 ユーザーワークフロー設計

### 2.8.1 初回セットアップフロー

```
1. ユーザー：「SFTP を設定」
       ↓
2. Claude: sftp-init.sh を実行
       ↓
3. スクリプト：対話式パラメータ入力
   - ホスト
   - ユーザー名
   - リモートパス
       ↓
4. ユーザー：秘密鍵を配置
       ↓
5. ユーザー：「秘密鍵をバインド」
       ↓
6. Claude: sftp-keybind.sh を実行
   - 自動で秘密鍵を検出
   - 権限を修正
   - 設定にパスを書き込み
       ↓
7. 完了：アップロード準備完了
```

### 2.8.2 通常使用フロー

```
1. ユーザー：コードを編集
       ↓
2. ユーザー：「コードをサーバーに同期」
       ↓
3. Claude: 変更ファイルを検出
   - コミット済み変更
   - 未コミット変更
   - 未管理ファイル
       ↓
4. Claude: SFTP バッチファイルを構築
       ↓
5. Claude: アップロードを実行
       ↓
6. 完了：「5 ファイルをアップロードしました」
```

### 2.8.3 エラー処理フロー

```
1. エラー：接続失敗
       ↓
2. 確認：
   - ネットワーク接続
   - .server アドレスとポート
   - 秘密鍵の権限
       ↓
3. ユーザーに明確なエラーメッセージを表示
       ↓
4. 修正アクションを提案
```
